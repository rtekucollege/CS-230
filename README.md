# CS-230

Briefly summarize The Gaming Room client and their software requirements. Who was the client? What type of software did they want you to design?
The Gaming Room is a game company that had an existing Android app called Draw It or Lose It - basically a drawing guessing game where teams compete against each other. They wanted to expand it beyond just Android so it could work across multiple platforms like web browsers, iOS, and desktop. The main requirements were that the game needed to support multiple teams with multiple players each, game and team names had to be unique, and only one instance of the game should exist in memory at a time. So the job was to design a server-side software architecture that could handle all of that across different devices.

What did you do particularly well in developing this documentation?
One thing I think I did pretty well in the documentation was explaining the design decisions in a way that actually made sense and connected back to the client's needs, rather than just listing technical stuff. Like when recommending Linux on AWS, I didn't just say “use Linux”, I explained why it made sense for their specific situation in terms of cost, security, and scalability. I also think the domain model section came out clear since breaking down how Game, Team, and Player all relate to each other through the Entity class made the structure easy to follow. 

What about the process of working through a design document did you find helpful when developing the code?
Working through the design document before touching any code was honestly more helpful than I expected. Having to think through things like how the singleton pattern would work for GameService, or how storage should be split between MySQL and S3, before actually implementing anything meant I had a clearer picture of what I was building. It basically forced me to catch potential issues at the planning stage instead of halfway through writing code when it's way more annoying to fix.

If you could choose one part of your work on these documents to revise, what would you pick? How would you improve it?
If I could go back and revise one part it would probably be the evaluation section where we compared the different operating platforms. I think I could have gone deeper there with more specific technical comparisons instead of keeping it fairly surface level. Adding more concrete details about things like actual performance differences or specific security features of each platform would have made that section a lot stronger and more useful for someone actually making that decision.

How did you interpret the user’s needs and implement them into your software design? Why is it so important to consider the user’s needs when designing?
Interpreting the user's needs was mostly about reading between the lines a bit. The client said they wanted to expand to other platforms, which meant the backend needed to be platform agnostic so REST APIs made sense because any device can talk to them regardless of what it's built with. The singleton requirement told me that data consistency was important to them even if they didn't use that exact term. Considering user needs in design is important because if you just build something technically impressive but it doesn't actually solve the client's real problems, the whole thing is kind of useless. The best code in the world doesn't matter if it's solving the wrong problem.

How did you approach designing software? What techniques or strategies would you use in the future to analyze and design a similar software application?
For approaching the design I started by identifying the core constraints like the singleton pattern, unique naming, multi-platform support and I let those drive the architectural decisions rather than just picking technologies that I was familiar with. Going forward I'd probably lean even more into this approach of starting with the constraints and requirements first before thinking about specific tools or technologies. I'd also use UML diagrams earlier in the process since having that visual map of how everything connects made it a lot easier to reason about the system as a whole before getting into the details.
